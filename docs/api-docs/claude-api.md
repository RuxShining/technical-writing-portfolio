# Overview

This sample API reference demonstrates the structure and level of detail commonly used in enterprise developer documentation.

The documentation focuses not only on request and response parameters, but also on operational behavior, retry handling, implementation considerations, and production integration scenarios.

## Audience

- Backend developers
- Platform engineers
- API integrators

# POST /v1/messages

Creates a conversational message request and returns Claude's response.

## Base URL

`https://api.anthropic.com`

# Authentication

All requests require API authentication.

## Required Headers

| Header | Required | Description |
| --- | --- | --- |
| `x-api-key` | Yes | Your Anthropic API key |
| `anthropic-version` | Yes | API version. Use `2023-06-01` |
| `content-type` | Yes | Must be `application/json` |
| `anthropic-beta` | No | Comma-separated beta feature flags |

# Request Body

## model – string, required

Specifies which Claude model processes the request.

| Model | Identifier | Recommended Use |
| --- | --- | --- |
| Claude Sonnet 4 | `claude-sonnet-4-20250514` | Balanced performance and speed |
| Claude Opus 4 | `claude-opus-4-20250514` | Complex reasoning tasks |
| Claude Haiku 4.5 | `claude-haiku-4-5-20251001` | Fast, lightweight requests |

## messages – array, required

Ordered conversation history.

Messages must:

- alternate between `user` and `assistant`
- start with a `user` message
- contain valid content blocks

### Message Object

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `role` | string | Yes | `user` or `assistant` |
| `content` | string or array | Yes | Text or structured content blocks |

## max_tokens – integer, required

Maximum number of output tokens generated by the model.

| Model | Maximum Tokens |
| --- | --- |
| Claude Sonnet 4 | 64,000 |
| Claude Opus 4 | 32,000 |
| Claude Haiku 4.5 | 8,096 |

## temperature – float, optional

Controls output variability.

| Value Range | Behavior |
| --- | --- |
| `0.0` | Deterministic output |
| `0.3–0.5` | Balanced technical output |
| `0.7–1.0` | More creative generation |

Default: `1.0`

## stream – boolean, optional

When enabled, responses are returned as Server-Sent Events (SSE).

Default: `false`

:::warning

Streaming connections may terminate before the final token is received due to:

- proxy timeout configuration
- interrupted client connections
- browser SSE limitations
- upstream network instability

Applications consuming streamed responses should:

- safely handle partial output
- avoid assuming completion without `stop_reason`
- implement retry logic where appropriate

:::

# Response Object

## 200 OK

| Field | Type | Description |
| --- | --- | --- |
| `id` | string | Unique message identifier |
| `type` | string | Always `message` |
| `role` | string | Always `assistant` |
| `content` | array | Content blocks |
| `model` | string | Model used |
| `stop_reason` | string | Why generation stopped |
| `usage` | object | Token usage metadata |

# Stop Reasons

| Value | Meaning |
| --- | --- |
| `end_turn` | Response completed normally |
| `max_tokens` | Output limit reached |
| `stop_sequence` | Custom stop sequence triggered |
| `tool_use` | Model is requesting tool execution |

# Error Handling

| HTTP | Error Type | Description |
| --- | --- | --- |
| 400 | `invalid_request_error` | Malformed request |
| 401 | `authentication_error` | Invalid or missing API key |
| 403 | `permission_error` | Insufficient API permissions |
| 429 | `rate_limit_error` | Rate limit exceeded |
| 500 | `api_error` | Internal server error |
| 529 | `overloaded_error` | Temporary service overload |

# Retry Recommendations

Retry logic should:

- use exponential backoff
- respect `Retry-After` headers
- avoid immediate parallel retries after `429`

Recommended retry pattern:

- initial retry: 1 second
- double delay after each failure
- maximum wait: 60 seconds

# Operational Notes

## Conversation Length

Long conversation histories significantly increase:

- token usage
- latency
- response cost

For long-running sessions:

- summarize older context
- trim redundant turns
- avoid replaying entire conversations unnecessarily

## Security Considerations

:::danger

Never:

- expose API keys in frontend applications
- commit credentials to repositories
- log sensitive customer prompts in plaintext

:::

Recommended:

- environment variables
- secret management tools
- request sanitization for logging systems
